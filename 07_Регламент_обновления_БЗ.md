# 07. Регламент обновления Базы Знаний — СКЗС

**Назначение файла:** Свод правил и процессов, по которым Проект СКЗС живёт и развивается. Кто, когда, как и зачем обновляет файлы БЗ. Как Claude сигнализирует о новом, как пользователь и эксперты согласовывают изменения, как менеджер физически вносит правки в Project.

**Аудитория:** Менеджер (физически обновляет файлы), администратор Project (это пользователь), эксперты (главный инженер, снабженец, главный сметчик).

**Ответственный за обновление этого файла:** Администратор Project (пользователь)
**Цикл обновления:** Редко — при изменении самих процессов работы
**Версия:** 1.0
**Дата последнего обновления:** 2026-05-18

---

## Часть 1. Принципы

### П.1. БЗ — живой документ
Файлы Базы Знаний не «замораживаются» после создания. Они **постоянно растут и уточняются** по мере появления новых объектов, обнаружения ошибок, изменения рыночных цен. Без регулярного обновления точность смет упадёт за 3–6 месяцев.

### П.2. Структура неприкосновенна
Иерархия 24 разделов сметы (16 из Шаблона + 8 согласованных новых) **не меняется без явного согласования**. Удалять разделы нельзя. Менять порядок нельзя. Добавлять новые разделы — только через явное согласование пользователя (см. часть 3).

### П.3. Каждое изменение требует согласования
Никакие правки в файлы БЗ **не вносятся молча**. Любое изменение проходит через одобрение пользователя в обучающем чате. Это защищает БЗ от деградации и обеспечивает прозрачность.

### П.4. У каждого файла один ответственный
За каждый файл отвечает один эксперт. Только он принимает решения о структуре и содержании. Остальные могут предлагать правки, но решение — за ответственным.

### П.5. Версионирование обязательно
В конце каждого файла — таблица «История версий» с датой и описанием изменений. После любого обновления — новая версия (1.0 → 1.1 → 1.2 → 2.0 при крупных изменениях).

### Роль каждого участника

**Главный инженер** — экспертиза по работам, методике, расценкам ФОТ. Утверждает все правки, связанные со стоимостью работ, формулами объёмов, структурными правилами.

**Снабженец** — экспертиза по материалам и арендным механизмам. Утверждает все правки цен на материалы, аренду техники, поставщиков.

**Главный сметчик** — экспертиза по структуре сметы и соответствию Шаблону СКЗС. Утверждает правки в файлах 04 и 05.

**Менеджер** — основной работник в Project. Делает расчёты, собирает обратную связь, физически обновляет файлы в Project, ведёт обучающий чат с Claude. Обучается администратором (пользователем).

**Администратор Project (пользователь)** — главный по проекту. Принимает решения о любых системных изменениях, согласовывает добавление новых разделов, утверждает финальные версии файлов. Контролирует качество работы.

## Часть 3. Цикл обновления БЗ

### Стандартный цикл — 7 шагов

```
1. Менеджер выполнил расчёт в рабочем чате Project
   ↓
2. Сметчик СКЗС сделал реальную смету по тому же объекту
   ↓
3. Менеджер открывает обучающий чат, выгружает:
   - Свой расчёт из рабочего чата
   - Реальную смету сметчика
   ↓
4. Claude в обучающем чате проводит сравнительный анализ:
   - Где расхождения
   - Почему расхождения
   - Какие правки нужны в БЗ
   ↓
5. Claude формулирует конкретные предложения:
   «Предлагаю обновить файл 01: добавить расценку X = Y руб/ед.
    Источник: <объект>. Подтверждаешь?»
   ↓
6. Менеджер согласовывает с пользователем (тобой) и/или
   профильным экспертом (инженер/снабженец/сметчик).
   ↓
7. После согласования Claude формирует обновлённую версию
   файла, менеджер заменяет файл в БЗ Project.
```

### Когда инициировать цикл

- **После каждых 2–3 новых разобранных объектов** (плановое обновление)
- **Не реже раза в 2 недели** (даже если новых объектов мало)
- **Срочно** — при сигналах Claude об отклонениях (см. часть 4)
- **Срочно** — при изменении рыночных цен поставщиками (≥10%)

## Часть 4. Сигналы об обновлении БЗ (5 триггеров)

Claude в **любом чате** Project (рабочем или обучающем) обязан сообщать менеджеру и пользователю, когда обнаруживает один из следующих триггеров:

### Триггер 1. Новый тип конструктива
**Что:** В проекте встречается материал или конструкция, которых нет в БЗ.
**Примеры:** СИП-панели, каркасник, дом из бруса, эксплуатируемая кровля-терраса, фундамент на винтовых сваях.
**Действие Claude:** Сообщить:
> «Встретилось новое: <описание>. Тип в БЗ отсутствует. После расчёта этого объекта рекомендую добавить новый раздел или подраздел. Источник: <название объекта>. Подтверждаешь?»

### Триггер 2. Расценка вне диапазона ±30%
**Что:** Фактическая цена (по итогам сравнения с реальной сметой) отличается от расценки в БЗ более чем на 30%.
**Примеры:** В БЗ кладка газоблока 300 мм = 7 000 ₽/м³, а в реальной смете нового объекта — 9 500 ₽/м³.
**Действие Claude:** Сообщить:
> «Расценка X в файле <имя> требует обновления. В БЗ: <старая цена>. Факт: <новая цена>. Разница: <%>. Источник: <объект>. Предлагаю обновить или добавить как альтернативу. Подтверждаешь?»

### Триггер 3. Новое системное правило (повтор 2+ раз)
**Что:** Обнаружено правило, которое работает на 2+ разных объектах, и сейчас его в файле 03 нет.
**Примеры:** «Если дом на склоне — нулевой цикл +30%», «Если дом из лицевого кирпича без штукатурки — облицовка дороже на X%».
**Действие Claude:** Сообщить:
> «Выявлено повторяющееся правило: <формулировка>. Подтверждено на объектах <список>. Предлагаю добавить в файл 03 раздел C. Подтверждаешь?»

### Триггер 4. Уникальная комбинация параметров
**Что:** Объект с сочетанием характеристик, которое не встречалось в эталонах.
**Примеры:** Двухэтажный дом с монолитным каркасом и плоской кровлей одновременно. Дом из газоблока с подвалом.
**Действие Claude:** Сообщить:
> «Объект имеет уникальную комбинацию: <описание>. Аналога в БЗ нет, точность расчёта может быть ниже стандартной. После получения реальной сметы рекомендую добавить этот объект в эталоны для будущих похожих случаев».

### Триггер 5. Расхождение по итогу >20%
**Что:** Сравнение расчёта Claude с реальной сметой показывает разницу больше 20% в итоге (а не только в отдельных позициях).
**Примеры:** Claude посчитал 5 млн, реальная смета 7 млн (+40%).
**Действие Claude:** Сообщить:
> «Расхождение >20% по итогу — требует разбора. Основные расхождения в разделах <список>. Возможные причины: <гипотезы>. Предлагаю детальный анализ для обновления нескольких файлов БЗ одновременно».

## Часть 5. Как Claude формулирует предложения о правках

### Формат предложения от Claude

Когда Claude в обучающем чате (или в рабочем — для срочных случаев) обнаруживает повод обновить БЗ, его сообщение должно соответствовать строгому формату:

```
ПРЕДЛОЖЕНИЕ ОБ ОБНОВЛЕНИИ БЗ

Файл: <имя файла>
Тип изменения: добавление новой позиции / правка существующей / новое правило
Раздел / подраздел: <если применимо>

Что предлагаю:
<конкретная формулировка изменения>

Источник:
<объект, факт, основание>

Влияние:
<на что повлияет изменение — точность каких будущих расчётов улучшится>

Запрос подтверждения:
Подтверждаешь? Если да — сформирую готовую обновлённую версию файла для замены в БЗ.
```

### Что пользователь делает с предложением

1. **Прочитать.**
2. Если правка простая (новая цена, очевидная по факту) → **подтвердить** «да, обновляй».
3. Если правка требует экспертизы (новое правило, спорная цена) → **отправить эксперту** (инженеру/снабженцу/сметчику) → получить от него заключение → подтвердить или отклонить.
4. Если правка несвоевременная или неверная → **отклонить** с объяснением, чтобы Claude учёл при следующих сигналах.

### Принцип: ничего не добавляется молча

Claude **не имеет права** молча обновлять файлы БЗ. Даже самая очевидная правка проходит через подтверждение в обучающем чате. Это страховка от деградации БЗ из-за единичных нетипичных случаев.

## Часть 6. Физическое обновление файла в Project

### Кто делает: менеджер

После того как Claude в обучающем чате выдал готовую обновлённую версию файла, менеджер физически заменяет старый файл новым в БЗ Project.

### Пошаговая инструкция (для менеджера)

1. Сохранить полученный от Claude файл на локальный диск с правильным именем (например, `01_База_расценок_ФОТ.xlsx`).
2. Открыть Project в Claude.
3. Перейти в раздел Files (Knowledge Base) правой панели.
4. Найти старый файл с таким же именем.
5. Удалить старый файл.
6. Загрузить новый файл.
7. Проверить, что файл действительно загрузился — открыть его в Project, посмотреть содержание.
8. Записать в локальный журнал (см. раздел 7): дата, какой файл обновлён, краткое описание правки, номер версии.

### Альтернатива: точечное редактирование (для .md файлов)

Для текстовых файлов (markdown) альтернативный способ — открыть файл, скопировать в текстовый редактор (Notepad++, VS Code), внести правки, сохранить, загрузить обратно. Удобно для мелких правок.

Для Excel-файлов (01 и 02) **всегда заменяется целиком**, точечное редактирование не делается.

### Что делать при ошибке

Если после обновления что-то пошло не так (Claude стал хуже считать, ссылается на старые правила и т.п.):
1. Восстановить предыдущую версию файла из локального архива (см. часть 7).
2. Сообщить пользователю.
3. Разобраться, что не так, в обучающем чате.

## Часть 7. Архивирование и страховка

### Локальный архив всех версий

**Менеджер хранит у себя на локальном диске** полную копию всех версий всех файлов БЗ. Папочная структура:

```
СКЗС_Project_Стройка/
├── текущая_версия/
│   ├── 01_База_расценок_ФОТ.xlsx
│   ├── 02_База_расценок_МАТ_МЕХ.xlsx
│   ├── ...
│   └── 11_Инструкция_для_администратора.md
├── архив/
│   ├── 2026-05-18_версия_1.0/
│   ├── 2026-06-01_версия_1.1/
│   └── ...
├── журнал_изменений.md
└── паспорт_Project.md
```

### Журнал изменений

Простой markdown-файл, в который менеджер записывает **каждое** изменение. Формат:

```markdown
# Журнал изменений БЗ Project «Стройка»

## 2026-05-25 — версия 1.1

**Файл:** 01_База_расценок_ФОТ.xlsx
**Изменение:** Добавлена расценка ФОТ для кладки СИП-панелей: 850 ₽/м²
**Основание:** Новый объект «ул. Берёзовая 12»
**Согласование:** Главный инженер подтвердил 24.05.2026
**Версия файла:** 1.0 → 1.1

---

## 2026-06-01 — версия 1.2
...
```

### Что делать при потере / порче Project

Если Project в Claude случайно удалён, файлы повреждены, или нужно создать копию для другой компании:

1. Открыть локальный архив, взять последнюю текущую версию всех файлов.
2. Создать новый Project в Claude по инструкции из «Паспорта Project».
3. Загрузить все файлы из локального архива в новый Project.
4. Восстановить Project Instructions из «Паспорта».
5. Время восстановления: 1–2 часа при наличии актуального архива.

**Без локального архива восстановление невозможно.** Это критически важная страховка.

## Часть 8. Регулярные ритуалы

### Раз в 2 недели — Цикл сверки

Менеджер вместе с пользователем:
1. Просматривают журнал изменений за период.
2. Проверяют, что все согласованные правки внесены.
3. Если за период не было обновлений — обсуждают, не нужно ли инициировать обновление по плановому циклу (2–3 объекта).

### Раз в месяц — Цикл рыночных цен

Снабженец:
1. Проверяет цены на 10 ключевых позиций материалов (бетон, арматура, кирпич, газоблок, цемент, песок, щебень, доска, фанера, утеплитель).
2. Если цена ушла больше чем на 10% — инициирует обновление файла 02 через обучающий чат.

### Раз в квартал — Цикл методики

Главный инженер:
1. Просматривает файл 03 (Системные правила) и файл 06 (Методика расчёта).
2. Решает, нужны ли уточнения или дополнения.
3. Проверяет, что новые накопленные правила (если есть) — отражены в файлах.

### Раз в полгода — Цикл структуры

Главный сметчик:
1. Просматривает файл 04 (Структура сметы).
2. Проверяет, соответствует ли структура реальной практике СКЗС.
3. Если за период появились новые типы работ — предлагает добавить новые разделы.

### Раз в полгода — Цикл формата

Менеджер + руководитель:
1. Просматривают файл 05 (Формат вывода).
2. Собирают обратную связь от клиентов СКЗС: что в смете нравится, что напрягает.
3. Корректируют формат по обратной связи.

### Раз в год — Цикл больших обновлений

Пользователь + все эксперты:
1. Полный обзор всей БЗ.
2. Решение о крупных архитектурных изменениях (если нужны).
3. Возможный переход на новую мажорную версию БЗ (1.x → 2.0).

## Часть 9. Что НИКОГДА не делается с БЗ

1. **Молчаливое обновление файлов** Claude — никогда. Все правки через явное подтверждение.
2. **Удаление эталонных объектов** из папки 09 — никогда. Эталоны — основа калибровки, их только дополняют.
3. **Изменение структуры верхнего уровня** (24 раздела) без согласия пользователя — никогда.
4. **Внесение правок одним участником в чужую зону ответственности** — никогда. Снабженец не правит ФОТ. Инженер не правит цены материалов.
5. **Замена файла без локального архивирования старой версии** — никогда. Сначала архивируем, потом меняем.
6. **Использование цен из внешних источников** в смете — никогда. Только из файлов 01 и 02 БЗ.
7. **Игнорирование сигнала Claude об отклонении** — никогда. Каждый сигнал должен быть рассмотрен и закрыт (подтверждён или отклонён с объяснением).
